CCM: what it is and how it works

  • What it is: a small Windows system monitor focused on CPU behavior (utilization, scheduling pressure, context switching) with a fast Direct2D/DirectWrite UI.
  • What it is not: a full “hardware sensor suite” like HWiNFO by default. Many sensors (package power, real CPU package temp, motherboard fans) are not reliably available from standard user-mode APIs.
  • How it works: CCM reads a mix of Windows APIs (PDH counters, ETW best-effort, PowrProf, WMI best-effort) on a periodic tick and renders a rolling history graph.
  • Best-effort rule: if a counter/provider is missing or blocked by policy/privileges, CCM shows N/A or 0 and keeps running.
If you want deeper interpretation, start with Metrics reference, then follow links into the deeper pages.

Quick tips

  • The UI runs continuously (~60 FPS) using Direct2D/DirectWrite.
  • Sampling for most counters happens ~4 times/sec; the history graph is a rolling window.
  • ETW: kernel tracing often requires Administrator. If ETW can’t start, ETW fields show as N/A/0 but the app still runs.
  • Pwr (W): many systems do not expose a PDH power meter. 0.0 W is common and usually means “no provider / not supported”.
  • Temp: WMI thermal zones frequently do not represent CPU package temperature; N/A is common.
